## 概念讲解

### 1. 实战案例：两大最有价值的 Agent 应用领域

Anthropic 在和客户合作中，发现两个领域的 Agent 应用最有实际价值。这两个领域有共同特征：

**Agent 最适合的任务特征**：
- 需要**对话 + 动作**结合（不是纯聊天，也不是纯执行）
- 有**清晰的成功标准**（能判断做得好不好）
- 支持**反馈循环**（能从结果中学习改进）
- 有**有意义的人类监督**（不是完全放手）

---

### 2. 案例 A：客服 Agent（Customer Support）

**为什么客服是 Agent 的天然场景**：

| 特征 | 客服场景的体现 |
|------|--------------|
| 对话 + 动作 | 一边和用户聊天，一边查数据、执行操作 |
| 成功标准 | 用户的问题是否解决（user-defined resolutions） |

**客服 Agent 使用的具体工具**：

原文将客服 Agent 的工具分为两类——信息获取类和动作执行类。这种区分很重要，因为它展示了"工具"这个抽象概念在实际场景中到底长什么样：

| 类别 | 具体工具 | 作用 |
|------|----------|------|
| 信息获取（Pull） | 拉取客户数据 | 了解客户是谁、什么级别 |
| 信息获取（Pull） | 查询订单历史 | 了解客户买了什么、有没有异常 |
| 信息获取（Pull） | 搜索知识库文章 | 找到标准处理方案 |
| 动作执行（Action） | 发起退款 | 直接解决客户的退款需求 |
| 动作执行（Action） | 更新工单 | 记录处理结果、流转到下一环节 |

这些工具不是假想的——它们是真实客服场景中 Agent 需要调用的接口。理解这些具体例子，才能在设计自己的 Agent 时知道"工具"到底该做成什么粒度。

**关键验证**：有些公司已经开始用**按解决量收费（usage-based pricing）**的模式——只有成功解决用户问题才收费。这说明他们对 Agent 的效果有足够的信心。

> **思考**：客服 Agent 是一个很好的"渐进式" Agent 场景——你可以从简单的 FAQ 路由（Workflow）开始，逐步增加工具和自主能力，最终变成一个完整的 Agent。

---

### 3. 案例 B：编程 Agent（Coding Agents）

**为什么编程是 Agent 的绝佳场景**：

| 特征 | 编程场景的体现 |
|------|--------------|
| 可验证 | 代码能通过自动化测试验证 |
| 反馈循环 | 测试结果就是反馈，Agent 可以基于测试失败信息修改代码 |
| 结构化 | 问题空间定义清晰（文件、函数、测试） |
| 可度量 | 输出质量可以客观衡量 |

**Anthropic 的实践**：他们的 Agent 已经能在 SWE-bench Verified 基准上，仅根据 pull request 描述就解决真实的 GitHub issue。

> **但是**：虽然自动化测试能验证功能正确性，**人类审查（human review）仍然不可或缺**——需要确保解决方案符合更广泛的系统需求。测试通过不等于代码好。

**编程 Agent 的能力进化路径**：

```
代码补全（Code Completion）
    ↓
代码生成（Code Generation）
    ↓
自主问题解决（Autonomous Problem-Solving）
```

### 4. 工具的 Prompt Engineering（Appendix 2 核心内容）

**核心观点**：工具定义（tool definitions）需要和整体 prompt 一样多的 prompt engineering 投入。

**什么是工具定义？** 在 Anthropic API 中，你通过指定工具的**结构和定义**来让 Claude 使用外部服务和 API。如果 Claude 决定要用某个工具，它会在响应中返回一个 tool use block。

---

### 5. 工具格式选择：同一动作的不同表示

同一个操作可以有不同的格式。选错格式会让 LLM 的表现大幅下降。

**例子**：文件编辑

| 格式 | 优点 | 缺点 |
|------|------|------|
| 写 diff | 精确、紧凑 | LLM 需要在写代码前就算出 chunk header（改了几行），这对 LLM 来说很难 |
| 重写整个文件 | LLM 容易写 | 冗余，Token 消耗大 |

**例子**：结构化输出

| 格式 | 优点 | 缺点 |
|------|------|------|
| Markdown 代码块 | 自然、LLM 见过很多 | 结构不如 JSON 严格 |
| JSON 包裹代码 | 结构化好 | 需要转义换行符和引号，LLM 容易出错 |

> **在软件工程中**，这些格式差异是"表面的"，可以无损转换。但**对 LLM 来说**，某些格式比其他格式难写得多。

**Anthropic 的三条格式选择建议**：

1. **给模型足够的 token 来"思考"**，再写出关键内容——避免让模型一开始就把自己逼入死角
2. **格式尽量接近模型在训练数据中自然见到的**——互联网文本里常见的格式 LLM 写起来更自然
3. **避免格式开销**——比如精确计算行数、字符串转义这类机械操作

### 6. Agent-Computer Interface (ACI) 设计

**核心类比**：你花多少精力设计 HCI（人-计算机接口），就应该花**至少同样多**的精力设计 ACI（Agent-计算机接口）。

**ACI 设计的四条指导原则**：

#### 原则 1：换位思考
> **把自己放在模型的位置**。仅凭工具的描述和参数，使用方式是否显而易见？如果你自己都要想半天，模型也会。

好的工具定义应该包含：
- 使用示例
- 边界情况说明
- 输入格式要求
- **和其他工具的清晰界限**——明确告诉模型"这个工具做什么，不做什么"，避免工具之间的功能重叠导致模型困惑

#### 原则 2：清晰命名
> **参数名和描述要不言自明**。就像给团队里的初级开发者写文档一样。

**特别提醒**：当你有很多**相似的工具**时，命名和描述的区分度就更关键了。原文原话："This is especially important when using many similar tools." 想象一下，如果你有 `search_docs`、`search_kb`、`search_faq` 三个工具，但描述都差不多，模型根本不知道该调哪个。相似工具越多，命名和描述就越需要精确区分。

#### 原则 3：经验性测试
> **用大量示例输入测试**模型如何使用你的工具。观察模型犯了什么错误，然后迭代改进。

使用 Anthropic 的 Workbench 可以方便地做这种测试。

#### 原则 4：防呆设计（Poka-yoke）
> **修改参数设计，让犯错变得更难**。

**Anthropic 的真实经验**：

> 在构建 SWE-bench Agent 时，我们在**优化工具上花的时间比优化整体 prompt 还多**。比如我们发现模型在使用相对路径时容易出错（Agent 移出了根目录后就会路径错误）。解决方案：把工具改成**只接受绝对路径**。改了之后模型使用完美，再没出过错。

这个例子完美说明了 Poka-yoke 的价值——**不是教模型不犯错，而是让犯错变得不可能**。

### 7. 总结：从 Workflow 到 Agent 的完整图景

```
┌─────────────────────────────────────────────────────────┐
│                  简单 → 复杂 决策路径                      │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  单次 LLM 调用 + RAG                                     │
│       ↓ 不够用                                           │
│  Prompt Chaining / Routing                              │
│       ↓ 需要并行                                         │
│  Parallelization                                        │
│       ↓ 子任务不可预测                                    │
│  Orchestrator-Workers                                   │
│       ↓ 需要迭代优化                                     │
│  Evaluator-Optimizer                                    │
│       ↓ 需要完全自主                                     │
│  Autonomous Agent                                       │
│                                                         │
│  关键原则：只在有可证明的改善时才升级复杂度                   │
└─────────────────────────────────────────────────────────┘
```

## 重点标记

1. **Agent 最适合的任务**：需要对话+动作、有清晰成功标准、支持反馈循环、有人类监督
2. **客服和编程是两大最有价值的 Agent 场景**——前者天然需要对话+操作，后者有完美的自动化验证
3. **客服工具的两类划分**：信息获取（拉客户数据、订单历史、知识库）和动作执行（发退款、更工单）——让"工具"从抽象概念变为具体接口
4. **工具定义 = Prompt Engineering**：花在工具设计上的精力应该和 prompt 一样多
5. **格式选择影响巨大**：避免 diff（需要预计算行数）和 JSON 包裹代码（需要转义）
6. **ACI 设计四原则**：换位思考、清晰命名、经验测试、防呆设计
7. **相似工具的命名陷阱**：工具越多越相似，命名和描述的区分度越重要
8. **Poka-yoke 实战案例**：绝对路径 vs 相对路径，改参数比教模型更有效
9. **测试通过 ≠ 代码好**：人类审查仍不可或缺

# v0 → v1 对比报告

> 本文档列出 v1 相对于 v0 的所有内容变更，供用户对比验收。
> 变更按模块组织，每处标注**变更类型**和**来源**。

---

## 变更类型说明

| 标记 | 含义 |
|------|------|
| **+新增** | v0 中完全没有的内容 |
| **↑强化** | v0 有提及但不够充分，v1 做了补充 |

---

## 模块一：基础理念与架构基石

### 变更 1：业界定义混乱的背景 [↑强化]
- **v0**：直接给出 Anthropic 的 Workflow vs Agent 分类
- **v1**：在定义之前增加了背景段落，说明业界对 "Agent" 定义没有共识（有人指完全自主系统，有人指预定义工作流），解释了为什么 Anthropic 需要先做分类
- **来源**：验证报告 #5（p1，中价值）
- **价值**：帮读者理解 Anthropic 做分类的动机，而不是凭空冒出一个分类

### 变更 2：Agent 的 "at scale" 限定 [↑强化]
- **v0**：表格中 Agent 写的是"需要灵活决策、规模化"
- **v1**：增加了一段专门解释为什么 "at scale" 很重要——Agent 的价值在大量持续的灵活决策场景才能体现，小任务用 Workflow 就够了
- **来源**：验证报告 #6（p2，中价值）
- **价值**：避免读者误以为任何需要灵活性的场景都该用 Agent

### 变更 3：框架的两阶段策略 [+新增]
- **v0**：建议"先裸写 API 调用，理解底层后再考虑框架"
- **v1**：新增原文总结部分的框架策略——"框架可以帮你快速开始，但到生产环境时要敢于减少抽象层"，明确区分了原型阶段和生产阶段的不同策略
- **来源**：验证报告 #2（p11，高价值）
- **价值**：比"框架要慎用"更精准——框架不是不能用，关键是知道什么时候该脱手

### 变更 4：Anthropic cookbook 资源引用 [+新增]
- **v0**：未提及
- **v1**：在框架建议部分提到 Anthropic 提供了 cookbook 示例实现
- **来源**：验证报告 #9（p2，低价值）
- **价值**：为想动手实践的读者提供了资源入口

### 变更 5：增强型 LLM 的前提假设 [+新增]
- **v0**：未提及这个前提
- **v1**：在模块一末尾明确指出"后续章节假设每个 LLM Call 都已具备增强能力"，并解释这意味着后面模式图中的每个方框都是增强型 LLM
- **来源**：验证报告 #1（p3，高价值）
- **价值**：**影响对整篇后续内容的理解**——读者不知道这个前提，可能误以为五种模式不需要 RAG/工具

## 模块二：五大工作流模式

### 变更 6：前提提醒 [+新增]
- **v0**：直接进入五种模式
- **v1**：在模式列表前增加一行前提提醒，说明所有 LLM Call 都是增强型的
- **来源**：承接变更 5
- **价值**：确保读者在阅读模式细节时记住这个前提

### 变更 7：Orchestrator-Workers "拓扑相似" 原文表述 [↑强化]
- **v0**：说"和 Parallelization 看起来很像"
- **v1**：补充了原文的精确措辞 "topographically similar"（拓扑结构相似）
- **来源**：逐页验证时发现
- **价值**：更精确地传达原文意思

## 模块三：自主 Agent 设计

### 变更 8：Agent 正在走向生产 [+新增]
- **v0**：未提及时代背景
- **v1**：新增原文的时代背景定位——"Agents are emerging in production as LLMs mature"
- **来源**：验证报告 #7（p8，中价值）
- **价值**：让读者理解这不是理论概念，而是正在发生的技术趋势

### 变更 9：可信环境的限定条件 [+新增]
- **v0**：列了"在可信环境中运行"但没有展开
- **v1**：专门解释了 "trusted environments" 的含义——Agent 运行的环境应该是受控的
- **来源**：验证报告 #3（p9，高价值的一部分）
- **价值**：强调了 Agent 部署的环境前提条件

### 变更 10：错误累积放大机制 [+新增]
- **v0**：重点标记中提到"错误可能累积"但没解释机制
- **v1**：新增独立小节"Agent 的风险：错误累积放大"，用多米诺骨牌类比解释了 compounding errors 的机制——早期小错在后续步骤层层放大
- **来源**：验证报告 #3（p9，高价值）
- **价值**：读者需要理解**为什么** Agent 比 Workflow 需要更多安全措施

### 变更 11：沙盒测试建议 [+新增]
- **v0**：重点标记提了一句但没有上下文
- **v1**：在错误累积部分详细说明了 Anthropic 的建议——在沙盒环境中大量测试 + 配合 guardrails
- **来源**：验证报告 #3（p9，高价值）
- **价值**：从"要做测试"到"具体怎么做测试"——沙盒 + guardrails 是明确的可执行建议

### 变更 12：工具设计的前向引用 [↑强化]
- **v0**：在模块三未明确引用 Appendix 2
- **v1**：在运行循环部分增加了对 Appendix 2 的前向引用，说明工具设计的重要性
- **来源**：验证报告中 ⚠️ 标记
- **价值**：帮助读者理解模块三和模块四的联系

## 模块四：实战应用与工具工程

### 变更 13：客服工具的具体清单与分类 [+新增]
- **v0**：在表格中有"拉取客户数据、订单历史、知识库文章"和"发退款、更新工单"，但混在一个简单表格里
- **v1**：新增独立的"客服 Agent 使用的具体工具"小节，将工具分为"信息获取类（Pull）"和"动作执行类（Action）"两类，每个工具单独列出并说明作用
- **来源**：验证报告 #4（p12，高价值）
- **价值**：让"工具"这个抽象概念变得可触摸——读者能看到在真实场景中工具到底做成什么粒度

### 变更 14：相似工具的命名特别提醒 [+新增]
- **v0**：原则 2 提到清晰命名但没强调多相似工具场景
- **v1**：新增原文的特别提醒和具体例子（`search_docs`/`search_kb`/`search_faq`），说明相似工具越多命名区分度越重要
- **来源**：验证报告 #8（p13，中价值）
- **价值**：这是实战中的常见陷阱——很多人的工具命名不够精确导致模型选错工具

### 变更 15：工具边界的强调 [↑强化]
- **v0**：列了"和其他工具的明确界限"但没有解释
- **v1**：补充说明了为什么要明确界限——避免功能重叠导致模型困惑
- **来源**：验证报告中 ⚠️ 标记
- **价值**：让原则从"列表项"变为"可操作的建议"

## 概念讲解

### 核心问题：为什么 Agent 需要 Grounding？

LLM 的知识来自训练数据，有两个致命缺陷：
1. **过时**：训练数据有截止日期
2. **幻觉**：模型会"编造"看似合理但错误的信息

**Grounding（接地/落地）** 就是解决这个问题的——让 Agent 的回答基于**可验证的事实**，而不是"猜测"。

**三种 Grounding 技术的演进**：

```
RAG（基础） → GraphRAG（增强） → Agentic RAG（终极）
 语义匹配       理解关系              主动推理检索
```

---

### 1. RAG（Retrieval-Augmented Generation）— 基础检索增强

**核心机制**：先从外部知识库检索相关信息，再把检索结果作为上下文传给 LLM 生成回答。

```
用户提问 → 转为向量 → 在向量数据库中语义搜索 → 找到相关文档
                                                    ↓
                                    传给 LLM 作为上下文 → 生成基于事实的回答
```

#### 向量数据库：按"意思"搜索，不是按"关键词"

**传统搜索**：
```sql
SELECT * FROM products WHERE description LIKE '%适合宽脚%'
-- 结果：0条（因为数据库里没有这个精确短语）
```

**语义搜索（向量数据库）**：
```
查询："适合宽脚的鞋"
→ 能匹配到 "加宽版"、"宽松舒适"、"extra wide" 等语义相近的结果
```

#### 向量搜索的三步流程

1. **数据转向量**：用 ML 模型把文本/图片转成数字向量（embedding），语义相似的离得近
2. **存储和索引**：向量数据库存储这些向量并建专用索引
3. **查询匹配**：用户查询也转成向量，数据库找最近邻的向量返回

#### RAG 对 Agent 的好处

| 好处 | 说明 |
|------|------|
| 信息更新 | 检索到的信息比训练数据更新 |
| 更准确 | 大幅降低幻觉风险 |
| 响应更快 | 向量搜索速度极快 |
| 理解更全面 | 可以处理文本、图片等多种数据 |

**Google Cloud 方案**：Vertex AI Search（托管 RAG）、Vertex AI RAG Engine（数据框架）

#### RAG 的局限

RAG 把知识当作"扁平的、互不关联的事实集合"。对简单问答很好用，但**不理解数据之间的关系**。

---

### 2. GraphRAG — 知识图谱增强检索

**解决的问题**：RAG 只做"语义匹配"，不懂"概念之间的关系"。

**核心改进**：在 RAG 基础上加入知识图谱（Knowledge Graph），让 Agent 不仅知道相关的"片段"，还理解概念之间**怎么连接**。

**示例**：
```
普通 RAG：搜索"感冒治疗" → 返回几篇关于感冒的文章片段
GraphRAG：搜索"感冒治疗" → 理解 症状 → 原因 → 治疗方法 的关系链
```

**适用场景**：医疗 AI、法律分析等需要理解实体关系的复杂领域。

---

### 3. Agentic RAG — Agent 主动推理检索

**这是 Grounding 的最高级形式。**

**核心区别**：

| 普通 RAG | Agentic RAG |
|---------|-------------|
| Agent 被动接收检索结果 | Agent **主动规划**检索策略 |
| 一次搜索，拿到什么用什么 | 多步搜索，不满意就换策略再搜 |
| 适合简单问答 | 适合复杂的多跳查询 |

**机制**：Agent 使用 ReAct 框架来检索——分析复杂查询 → 制定多步计划 → 执行多次工具调用 → 综合最佳信息。

#### 实战示例：实时库存查询

```
1. 客户问："新款 SolarFlare 跑鞋有货吗？"

2. Agent 识别产品（语义搜索）:
   → 即使客户描述模糊，也能通过向量数据库匹配到正确产品

3. Agent 触发实时操作（函数调用）:
   → 调用 check_inventory(product_id) 查实时库存

4. Agent 回答:
   "有！SolarFlare 跑鞋现在有82双库存。"
```

**关键洞察**：Agentic RAG = 检索能力 + 行动能力。Agent 不仅能"查到"信息，还能"执行"操作。

#### Retrieve and Re-rank（检索重排序）

**进阶优化策略**：
1. 先"宽"检索：配置 Vertex AI Vector Search 返回比需要更多的候选文档
2. 再"精"重排：用 LLM 或专用 re-ranking 服务从候选中挑出最相关的

这解决了"召回率 vs 精确度"的经典矛盾。

### 4. 其他 Grounding 方法

除了 RAG 系列，Vertex AI 还提供：

| 方法 | 用途 |
|------|------|
| Grounding with Google Search | 连接世界知识，覆盖广泛话题 |
| Grounding with Google Maps | 基于地理数据的上下文感知回答 |
| Grounding with your data | 用 RAG 连接你的网站或文档集 |
| Grounding with Elasticsearch | 与现有 Elasticsearch 索引结合 |

**Google Search Grounding 的流程**：
```
用户提问 → Gemini 分析是否需要搜索 → 自动生成搜索查询
→ 执行搜索 → 处理搜索结果 → 生成带引用源的回答
```

## 重点标记

1. **RAG 是对抗幻觉的基础武器**：让 Agent 的回答有据可查
2. **三级演进**：RAG（语义匹配）→ GraphRAG（理解关系）→ Agentic RAG（主动推理）
3. **向量数据库 vs 传统数据库**：按"意思"搜索 vs 按"关键词"搜索
4. **Agentic RAG = 检索 + 行动**：Agent 不只搜信息，还能执行操作
5. **Retrieve and Re-rank**：先宽后精，解决召回率和精确度的矛盾
6. **Google Search Grounding 是最快的落地方式**：不需要自建知识库就能获得实时信息

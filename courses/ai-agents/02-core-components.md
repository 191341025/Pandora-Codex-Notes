## 概念讲解

每个 AI Agent 由四大核心组件构成：

```
┌────────────────────────────────────────────┐
│                 AI Agent                    │
│                                            │
│   ┌──────────┐  ┌──────────┐              │
│   │  Model   │  │  Tools   │              │
│   │  (大脑)   │  │  (四肢)   │              │
│   └────┬─────┘  └────┬─────┘              │
│        │              │                    │
│   ┌────┴──────────────┴─────┐              │
│   │    Orchestration         │              │
│   │    (执行功能/指挥官)      │              │
│   └────────────┬─────────────┘              │
│                │                            │
│   ┌────────────┴─────────────┐              │
│   │    Runtime / Memory       │              │
│   │    (运行环境/记忆系统)     │              │
│   └──────────────────────────┘              │
└────────────────────────────────────────────┘
```

---

### 1. Models（模型）— Agent 的大脑

#### 如何选择模型

**核心原则**：不是选最强的，而是选**最适合的**。

模型选择是三个维度的博弈：

```
能力 (Capability)
  ↑
  │        ← 能力越强，成本和延迟越高
  │
  └──────────→ 成本 (Cost) / 延迟 (Latency)
```

**最常见的错误**：场景不需要那么强的模型，却用了最贵的，导致钱花多了、速度还慢了。

#### 按场景选模型

| 场景 | 推荐模型 | 理由 |
|------|---------|------|
| 早期原型/批量简单任务 | Gemini 2.5 Flash-Lite | 最省钱最快，擅长翻译、分类 |
| 高流量生产应用 | Gemini 2.5 Flash | 平衡质量/成本/速度 |
| 复杂多步推理/前沿代码生成 | Gemini 3 Pro | 最强推理，PhD 级别 |

**多 Agent 架构的优势**：不同 Agent 可以用不同模型。复杂推理用重量级模型，日常查询用轻量级模型。这样**系统整体**的成本和性能都是优化的。

#### 模型调优（Fine-tuning）

用你自己的高质量数据集来"专业化"模型的知识和风格。

**注意区分**：
| 概念 | 作用 | 类比 |
|------|------|------|
| Fine-tuning | 调整模型的风格和专业知识 | 给医生进修特定专科 |
| Grounding | 连接实时数据源保证事实准确 | 给医生一本最新的诊疗指南 |

> Fine-tuning 改的是"怎么说"，Grounding 改的是"说什么"。

#### 推理 Token 控制

可以给特定调用分配更多推理 token，让模型花更多"算力"来深度思考。这是一种**以延迟和成本换准确率**的策略。

### 2. Tools（工具）— Agent 的四肢

**定义**：工具是 Agent 超越模型原生能力的"手脚"——让它能查数据库、调 API、执行代码。

**工具的种类**：
- **内部函数**：你团队写的业务逻辑
- **API 调用**：内部/外部服务接口
- **数据源**：数据库、向量存储等
- **其他 Agent**：在多 Agent 系统中，一个 Agent 可以把另一个当"工具"使用

---

### 3. 数据架构 — Agent 的记忆系统

Agent 需要三种不同类型的"记忆"，对应不同的数据服务：

#### 层级一：长期记忆（Long-term Knowledge Base）

像人的"长期记忆"——存储持久的知识和历史。

| 服务 | 角色 | 创业公司用例 |
|------|------|------------|
| **Vertex AI Search** | 语义搜索知识库 | 搜索产品文档、客服记录、论坛帖子 |
| **Firestore** | 用户状态和任务进度 | 存储用户引导流程的实时状态 |
| **Vertex AI Memory Bank** | 自动提取对话中的关键事实 | "用户偏好直飞"、"用户的狗叫 Fido" |
| **Cloud Storage** | 原始文件存储 | PDF、图片、音频等原始数据 |
| **BigQuery** | 分析型数据仓库 | "上周新功能的用户参与度如何？" |

#### 层级二：工作记忆（Working Memory）

像人的"短期记忆"——当前对话的上下文，必须**超低延迟**。

| 服务 | 角色 | 用例 |
|------|------|------|
| **Memorystore** | 亚毫秒级内存缓存 | 缓存高频/昂贵的工具调用结果，减少延迟和成本 |

#### 层级三：事务记忆（Transactional Memory）

像人的"操作日志"——记录关键操作，要求强一致性。

| 服务 | 角色 | 用例 |
|------|------|------|
| **Cloud SQL** | 关系型数据库 | Agent 执行的每个关键操作（付款、开通服务）都记录审计日志 |
| **Cloud Spanner** | 全球分布式数据库 | 当用户遍布全球时，从 Cloud SQL 升级 |

### 4. Orchestration（编排）— Agent 的指挥官

**定义**：编排是引导 Agent 完成多步骤任务的"执行功能"——决定用什么工具、什么顺序、怎么组合结果。

#### ReAct 框架（核心编排模式）

```
┌─→ Reason（推理）: 评估目标和当前状态，决定下一步
│       ↓
│   Act（行动）: 选择并调用合适的工具
│       ↓
│   Observe（观察）: 接收工具输出，整合到上下文
│       ↓
└── 继续循环，直到任务完成
```

#### 完整示例：处理退款

```
1. Reason: 用户要退款，先查退款政策
   → Act: semantic_search("refund policy")
   → Observe: "30天内全额退款"

2. Reason: 需要知道购买日期，去 CRM 查
   → Act: get_order_details(user_id)
   → Observe: purchase_date: '2025-07-20'

3. Reason: 今天7月29日，距购买9天，在30天内，可以退
   → Act: process_refund(order_id, amount)
   → Observe: status: 'success'

4. Final: "退款已处理，3-5个工作日到账。"
```

**编排的实际应用场景**：
- **自动客户引导**：创建账户 → 发欢迎邮件 → 检查用户是否完成首次操作
- **主动系统监控**：收到告警 → 查日志 → 重启 Pod → 通知 Slack
- **复杂销售线索分析**：查公司数据 → 查 CRM → 分配给高级销售或加入培育序列

### 5. Runtime（运行时）— Agent 的部署环境

Agent 从原型到生产需要的基础设施：

| 核心能力 | 说明 |
|---------|------|
| 可伸缩性 | 自动扩缩容，从零到百万请求 |
| 安全性 | 身份管理、网络访问控制、TLS 加密 |
| 可靠性与可观测性 | 错误处理、自动重试、日志、监控 |

**三种部署选项**：

| 选项 | 适合谁 | 特点 |
|------|--------|------|
| **Vertex AI Agent Engine** | 种子轮小团队 | 全托管，几天内从原型到生产 |
| **Cloud Run** | 快速增长的创业公司 | Serverless，按用付费，应对流量高峰 |
| **GKE** | B轮+有平台工程团队 | 最大控制力，融入现有 K8s 体系 |

## 重点标记

1. **选模型的核心原则是"够用就好"**：不是越强越好，是成本-速度-能力的最优平衡
2. **多 Agent 用多模型**：重活用大模型，轻活用小模型，系统级优化
3. **三层记忆架构很关键**：长期/工作/事务，分别用不同数据服务
4. **ReAct 是编排的核心**：Reason → Act → Observe 循环，直到任务完成
5. **运行时选择取决于团队阶段**：早期用 Agent Engine，成长期用 Cloud Run，成熟期用 GKE
6. **Fine-tuning ≠ Grounding**：一个改风格，一个保准确

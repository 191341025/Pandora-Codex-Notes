## 概念讲解

### 1. 两种文档使用方式：附加 vs 嵌入

在 AnythingLLM 中使用文档，第一个决定就是：**全文塞给 LLM 看，还是让系统检索相关片段？**

| 维度 | 附加文档 (Attached) | 嵌入文档 (Embedded / RAG) |
|------|---------------------|--------------------------|
| 工作原理 | 将文档**全文**塞进上下文窗口 | 将文档切片→向量化→按语义检索**片段** |
| 信息完整度 | **完整**——LLM 能看到全部内容 | **部分**——只取最相关的 4-6 个片段 |
| 作用域 | 当前 Thread 内有效 | 整个 Workspace 所有 Thread 可用 |
| 操作方式 | 拖拽到聊天窗口 / 点击回形针图标 | 在 Workspace 管理界面 "Move to workspace" |
| 速度与成本 | 慢、贵（全文消耗 Token） | 快、省（只检索几个片段） |
| 准确性 | 高（全文可见，不会漏信息） | 取决于切片质量和检索精度 |
| 适合场景 | 短文档、需要全文理解 | 长文档、大量文档、日常查询 |

怎么选？一句话：**能放进 Context Window 就附加，放不下就嵌入**。AnythingLLM 会在附加文档超出 Context Window 时自动提醒你。

附加文档就像你把整本书摊开放在桌上让 LLM 看——全都能看到，但桌子大小有限（Context Window），放不下太多书。嵌入文档就像把书拆成一页页存进图书馆的索引系统——你问问题时，图书管理员帮你找出最相关的几页拿给 LLM。看不到全书，但能处理海量藏书。

### 2. RAG 如何工作：跟着一个问题走完全程

假设你上传了一份产品手册，问"退货政策是什么？"——从上传文档到看到回答，背后发生了什么？

RAG（Retrieval Augmented Generation，检索增强生成）的全过程分两个阶段：先准备，再查询。

#### 准备阶段：让文档变成可检索的

**上传文档**

你把 PDF 拖进 AnythingLLM，系统把它转成纯文本。到这一步，**文档只是"存在"于系统中，LLM 完全看不到它**。

> 这是最常见的误解："我上传了文档，LLM 就应该知道了"——不是这样。上传只是转文本，必须执行下一步。

**嵌入文档**

你点击 "Move document to workspace"，触发三件事：
1. 纯文本被**切片（Chunking）**——拆成一段段小片段
2. 每个片段经过 **Embedding 模型**转换成一串数字（向量）
3. 向量存入**向量数据库**

到这里，文档才真正变成了可检索的状态。

> **这一步的固有局限**：切片过程中，相关内容不一定保持在一起。如果一个答案跨越了两个切片的边界，后面的检索可能只找到其中一个。这是当前学术界仍在活跃研究的问题。

#### 查询阶段：从问题到答案

**检索（Retrieval）**

你输入"退货政策是什么？"，系统做了三件事：
1. 用**同一个 Embedding 模型**把你的问题转成向量
2. 向量数据库用 **Cosine Distance**（余弦距离）计算问题向量与所有文档向量的"距离"
3. 返回距离最近的几个 Chunk，过滤掉低分结果

这里需要建立一个关键认知：**这个过程不是语义理解，是数学计算**。向量数据库不"懂"你在问什么——它在算数字之间的距离。所以措辞很重要：用文档中的原始词汇提问（"退货政策"而非"东西不想要了怎么办"），效果远好于用自己的话改述。

**生成（Generation）**

检索到的文本片段被拼接成 System Message，连同其他信息一起送入 LLM：

```
[System Prompt]          ← 你在 Workspace 设置的系统提示
[检索到的文档片段]         ← RAG 找到的相关内容
[对话历史]                ← 之前的问答
[用户当前问题]             ← "退货政策是什么？"
```

LLM 综合以上信息生成回答。注意：LLM **不知道**片段来自哪个文件，也不知道原始位置——它只看到一段拼接好的文本。

#### 完整流程一览

```
上传文档（文件 → 纯文本）
   ↓
嵌入文档（文本 → 切片 → 向量化 → 存入向量数据库）
   ↓                    ← Embedding 模型在这里工作
   ↓  ── 准备阶段结束 ──
   ↓
用户输入问题
   ↓
问题向量化（同一个 Embedding 模型）
   ↓
向量数据库计算相似度，返回最近的 Chunk
   ↓                    ← 相似度阈值在这里过滤
相关片段作为 System Message 注入 LLM
   ↓
LLM 综合所有信息 → 生成回答
```

> **常见误用**：有人上传了几百个菜谱，然后问"帮我找出 3 个高热量的菜"——这注定失败。RAG 是按语义相似度检索，不是全文扫描。需要全局分析（统计、排序、比较）时，应该用 Agent（模块 03），而不是 RAG。

### 3. 当 RAG 结果不理想：诊断与调优

理解了 RAG 流水线，你就能系统性地排查问题——关键是**定位问题出在哪一步**，然后用对应的参数去调。

#### 从流水线定位问题

| 现象 | 问题出在哪一步 | 排查方法 |
|------|--------------|---------|
| LLM 完全不知道文档内容 | 上传后没嵌入 | 检查文档是否已 "Move to workspace" |
| 检索到的内容不相关 | 检索阶段（措辞差异） | 用文档中的原始词汇重新提问 |
| 明明有相关内容但返回空 | 检索阶段（阈值过严） | 相似度阈值调至 "No Restriction" |
| 回答不完整，只有部分信息 | 生成阶段（片段太少） | 增大 Max Context Snippets |
| 答案跨越切片边界被截断 | 嵌入阶段（切片局限） | 更具体的问题 / Document Pinning |
| 非英文文档效果差 | 嵌入 + 检索（模型不适配） | 换用多语言 Embedding 模型 |
| Query 模式说找不到 | 文档确实没有该信息 | 切换到 Chat 模式让 LLM 用通用知识补充 |

#### 三个关键调优参数

**相似度阈值（Document Similarity Threshold）**

过滤低分向量——相似度低于阈值的 Chunk 不会被送给 LLM。默认 20%。

```
阈值高 → 只返回高度相关的片段 → 精确但可能漏掉有用信息
阈值低 → 返回更多片段 → 全面但可能引入噪音
No Restriction → 返回所有匹配 → 最全面，适合排查问题
```

影响阈值效果的三个因素：
- **Embedding 模型能力**：默认 `all-MiniLM-L6-v2` 对英文好，对中文打折
- **向量密度**：Workspace 中文档越多，噪声匹配越多
- **提问质量**：模糊问题得到模糊匹配——提问越具体，高分匹配越多

> **实用建议**：遇到"找不到相关信息"，第一步调到 "No Restriction"。有结果了再逐步收紧找平衡点。

**最大上下文片段数（Max Context Snippets）**

决定 RAG 最多取几个 Chunk 送给 LLM。推荐 4-6 个。

不是越多越好——Context Window 是共享的（System Prompt + 文档片段 + 历史 + 问题都在里面），太多片段会触发自动裁剪，过多"噪音"反而误导 LLM。大 Context Window 模型（如 Claude 200K）可以适当增大，但噪音问题依然存在。

**搜索偏好 / 重排序（Search Preference / Reranking）**

在默认向量检索基础上增加一轮重排序：先检索更多 Chunk，再用 Reranking 模型重新排序，取最相关的送给 LLM。约增加 100-500ms 延迟，首次使用时自动下载模型。

> 目前仅在使用 LanceDB（默认向量数据库）时可用。这是 Workspace 级设置，可以在不同 Workspace 分别实验。

#### Document Pinning：当检索不够用时的后备方案

调了上面的参数还不理想？还有一个策略——把关键文档"钉"在 Context Window 中，跳过 RAG 检索，直接全文注入。

```
普通嵌入文档：问题 → 向量检索 → 取片段 → 送给 LLM
钉选文档：    问题 → 直接全文注入 → 送给 LLM（跳过检索）
```

使用条件：
- 文档必须**先被嵌入**到 Workspace
- 全文必须**能放进 Context Window**
- 被 Pin 的文档**不参与 RAG 检索**——不会重复出现

**最佳策略**：Pin 1-2 个核心文档（如产品规格书、公司 SOP）+ 其余文档走 RAG——兼顾准确性和灵活性。不要 Pin 太多——10 个文档基本就把 Context 占满了。

#### Context Window 溢出处理

当附加文档超出 Context Window 时，AnythingLLM 给你三个选项：

| 选项 | 行为 | 建议 |
|------|------|------|
| **Cancel** | 移除该文档 | 安全选择 |
| **Continue Anyway** | 强行塞入，超出部分被裁剪 | **不推荐**——数据丢失，行为不可预测 |
| **Embed** | 转为嵌入文档（走 RAG） | **推荐**——自动解决溢出 |

> 鼠标悬停在聊天输入框的回形针图标上可查看当前 Context 使用情况。经常溢出说明你的文档不适合附加方式——直接用 RAG。

### 4. Embedding 模型与向量数据库的选型影响

这两个组件在模块 01 已介绍过选型维度，这里聚焦它们对 RAG 效果的具体影响。

**Embedding 模型**：默认 `all-MiniLM-L6-v2`（384 维，英文优化）。对中文、日文等效果会下降。更换模型后**所有已嵌入文档必须重新嵌入**——不同模型产出的向量在不同"语义空间"中，无法混用。这是系统级配置，所有 Workspace 共用。

**向量数据库**：默认 LanceDB 对大多数用户够用。需要切换的场景：百万级以上文档量、企业级高可用需求、特定查询功能（如 Metadata 过滤）。

> **新手建议**：先用默认配置跑通流程，发现具体问题再调整。过早优化这两个组件是最常见的时间浪费。

---

## 重点标记

1. **附加文档是全文注入，嵌入文档是片段检索**：前者准确但受 Context Window 限制，后者能处理大量文档但可能漏信息
2. **RAG 是数学过程，不是语义理解**：Cosine Distance 计算距离，不"懂"你的问题——措辞越接近原文效果越好
3. **上传 ≠ 可用**：文档上传后必须"嵌入到 Workspace"才能被 RAG 检索
4. **相似度阈值是第一个调的旋钮**：遇到"找不到相关信息"，先调到 "No Restriction"
5. **Pin 核心文档 + 其余走 RAG**：兼顾准确性和灵活性的最佳策略

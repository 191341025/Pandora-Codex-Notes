## 概念讲解

### 1. AnythingLLM 是什么

**定义**：AnythingLLM 是一个开源的一站式 AI 应用平台，把 RAG（检索增强生成）、AI Agent、文档对话、嵌入式聊天组件等能力集成在一个产品里，支持本地部署，无需写代码。

**核心思想**：一个平台，三大组件任意组合，数据不出本地。

**为什么重要/有效**：

传统上，搭建一个"用自己的文档和 AI 对话"的系统，你需要分别解决：
- 选哪个 LLM？怎么接入？
- 文档怎么切片？用什么 Embedding 模型？
- 向量数据库选哪个？怎么部署？
- 前端界面怎么做？权限怎么管？

AnythingLLM 把这些全部封装成一个产品，**每个组件都可以独立替换**：想用 OpenAI 就选 OpenAI，想用本地 Ollama 就换 Ollama，向量库想从 LanceDB 换到 Pinecone 也行——组件之间互不绑定。

**适用场景**：
- 个人用户：想在本地跟自己的文档对话，不想数据上传到第三方
- 企业团队：需要一个带权限管理的内部 AI 知识库
- 开发者：想快速原型验证 RAG 方案，不想从零搭建

> **与其他工具的区分**：ChatGPT 是一个固定 LLM 的在线服务；LangChain 是一个开发框架（需要写代码）；AnythingLLM 是一个**开箱即用的应用**，面向终端用户，组件可插拔。

### 2. Workspace（工作区）

**定义**：Workspace 是 AnythingLLM 的核心组织单元，类似于 ChatGPT 中的"对话"，但功能更强——每个 Workspace 可以有独立的文档、独立的 LLM 配置、独立的 System Prompt。

**直觉建立**：

把 AnythingLLM 想象成一栋办公楼。整栋楼共享基础设施（水电网络 = 系统级 LLM 和向量数据库），但每间办公室（= Workspace）可以有自己的装修风格（= System Prompt）、自己的资料柜（= 文档集合）、甚至可以单独换一台打印机（= Workspace 级 LLM 覆盖）。

你走进哪间办公室，就在哪间的上下文里工作。

> **类比边界**：办公室之间是完全隔离的——Workspace A 的文档不会出现在 Workspace B 的搜索结果里。

**为什么重要/有效**：

Workspace 隔离让你可以为不同任务创建专属环境：
- 一个 Workspace 放技术文档，System Prompt 设为"你是技术顾问"
- 另一个 Workspace 放合同模板，System Prompt 设为"你是法务助手"
- 两者互不干扰，LLM 不会把合同内容混入技术回答

**示例**：

```
Workspace: "产品文档助手"
├── 文档: product-spec-v3.pdf, api-reference.pdf
├── System Prompt: "你是产品文档专家，只基于上传的文档回答"
├── Chat Mode: Query（严格基于文档）
└── LLM: 使用系统默认 (GPT-4o)

Workspace: "创意头脑风暴"
├── 文档: (无)
├── System Prompt: "你是创意顾问，帮我发散思维"
├── Chat Mode: Chat（允许使用通用知识）
└── LLM: 覆盖为 Claude Sonnet（更有创意）
```

### 3. Query 模式 vs Chat 模式

**定义**：AnythingLLM 提供两种对话模式，决定了 LLM 回答时"参考什么信息"：

| 维度 | Query 模式 | Chat 模式 |
|------|-----------|----------|
| 信息来源 | **仅**参考上传的文档 | 文档 + LLM 自身的通用知识 |
| 找不到相关信息时 | 明确告诉你"没有找到相关信息" | 会用通用知识尝试回答 |
| 幻觉风险 | 低（宁可不答也不编） | 中（可能混入非文档内容） |
| 适合场景 | 技术文档查询、合规问答 | 头脑风暴、开放讨论 |

**直觉建立**：

Query 模式像一个**严谨的图书管理员**——你问他一个问题，他只会去书架上找书来回答。如果书架上没有相关的书，他会直接说"我们图书馆没有这方面的资料"，绝不会编造答案。

Chat 模式像一个**博学的朋友**——他也会先看你给他的资料，但如果资料里没有，他会结合自己的知识聊下去。更灵活，但你需要自己判断哪些是资料里的，哪些是他"自由发挥"的。

**为什么重要/有效**：

选错模式是新用户最常踩的坑：
- 用 Chat 模式做合规查询 → 可能得到 LLM 编造的"看起来很专业"但文档里没有的答案
- 用 Query 模式做头脑风暴 → 反复得到"没有找到相关信息"，体验很差

**常见问题与解决**：

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| Query 模式反复说"No relevant information found" | 1. 提问措辞与文档措辞不匹配<br>2. 相似度阈值太严格<br>3. 文档切片方式不佳 | 1. 用文档中的原始词汇提问<br>2. 将相似度阈值调到 "No restriction"<br>3. 尝试更具体的问题 |
| Chat 模式回答了但内容不准确 | LLM 用通用知识补充了文档中没有的内容 | 切换到 Query 模式，或提示 LLM "只根据提供的文档回答" |

**适用场景**：
- **先用 Query**：当你需要事实性、可溯源的回答时
- **切到 Chat**：当你需要解释、举例、延伸讨论时
- **调阈值**：如果 Query 模式找不到信息，先把相似度阈值调到 "No restriction" 试试

### 4. 功能全景：三大支柱 + 扩展能力

**定义**：AnythingLLM 的功能体系可以分为三大核心支柱和若干扩展能力。

**三大核心支柱**：

| 支柱 | 做什么 | 后续模块 |
|------|--------|---------|
| **RAG 文档对话** | 上传文档 → 切片 → 向量化 → 语义搜索 → LLM 生成回答 | 模块 02 |
| **AI Agent** | 给 LLM 配备工具（搜索、爬网页、写文件、查数据库等），让它执行多步任务 | 模块 03 |
| **Agent Flow** | 无代码可视化编排 Agent 工作流，拖拽积木式构建自动化流程 | 模块 04 |

**扩展能力**：

| 能力 | 说明 |
|------|------|
| 嵌入式聊天组件（Chat Widget） | 把 AnythingLLM 的对话窗口嵌入你自己的网站 |
| MCP 协议支持 | 通过 Anthropic 的 MCP 协议接入外部工具 |
| 开发者 API | 完整的 REST API，可以用代码管理 Workspace、上传文档、发起对话 |
| 多用户权限 | Admin / Manager / Default 三级角色（Docker 版专属） |
| 外观定制 | 自定义 Logo、欢迎语、页脚链接（Docker 版专属） |
| System Prompt Variables | 在 System Prompt 中注入动态变量（日期、用户名等） |

### 5. 聊天界面交互

**定义**：聊天界面是你与 AnythingLLM 交互的主战场，分为用户消息区、LLM 回复区和输入控制区三部分。

**用户消息操作**：

| 操作 | 功能 |
|------|------|
| Copy | 复制消息内容 |
| Edit | 修改并重新提交（**注意**：会截断该消息之后的所有对话历史） |
| Speak | 使用系统 TTS / OpenAI Voice / 11Labs 朗读 |

**LLM 回复操作**：

| 操作 | 功能 |
|------|------|
| Copy | 复制回复内容 |
| Edit | 修改 LLM 的输出内容（不会重新提交 prompt，仅更新历史记录） |
| Regenerate | 用相同 prompt 重新生成回答 |
| 反馈（👍/👎） | 标记回答质量。**不影响后续对话**，但导出 Chat Log 时可用于筛选高质量回答 |

**输入控制**：

| 控件 | 说明 |
|------|------|
| Slash Commands | 快捷文本注入，如 `/reset` 清空对话 |
| `@agent` | 在 prompt 开头加 `@agent` 启动 Agent 会话 |
| 字体大小 | 调整显示字号 |
| 麦克风 | 语音转文字输入（仅 Docker 版） |

> **实用技巧**：用户消息的 Edit 和 LLM 回复的 Edit 行为不同。前者会"从这个点重新来过"（截断后续），后者只是"修正这一条回复的记录"。

### 6. System Prompt Variables（系统提示变量）

**定义**：允许你在 Workspace 的 System Prompt 中使用 `{变量名}` 占位符，运行时自动替换为实际值。

**内置变量**：

| 变量 | 说明 | 可用版本 |
|------|------|---------|
| `{date}` | 当前日期 | 所有版本 |
| `{time}` | 当前时间 | 所有版本 |
| `{datetime}` | 当前日期时间 | 所有版本 |
| `{user.name}` | 用户名 | Docker 多用户模式 |
| `{user.bio}` | 用户简介 | Docker 多用户模式 |
| `{os.name}` | 操作系统名称 | Desktop 版 |
| `{os.arch}` | 系统架构 | Desktop 版 |

**自定义变量**：可以在设置页创建静态变量（如公司名称），在 System Prompt 中用 `{company_name}` 引用。

**示例**：

System Prompt 模板：
```
你是 {company_name} 的内部知识助手。
今天是 {date}，当前时间 {time}。
当前用户：{user.name}
用户背景：{user.bio}
请根据用户背景调整回答的技术深度。
```

运行时展开为：
```
你是 Acme Corp 的内部知识助手。
今天是 2026-02-20，当前时间 14:30:00。
当前用户：张三
用户背景：后端开发工程师，主要用 Python 和 Go
请根据用户背景调整回答的技术深度。
```

**适用场景**：
- 多用户环境下给每个人个性化的 AI 助手体验
- 让 LLM 知道当前时间（默认情况下 LLM 不知道"今天几号"）
- 企业部署时注入公司特定信息，无需每次手动写

> **注意**：无效变量不会报错，只是不展开。UI 中可以通过蓝色高亮判断变量是否有效。

## 重点标记

1. **Workspace 是隔离的**：文档、System Prompt、甚至 LLM 选型都可以按 Workspace 独立配置，不同 Workspace 之间的文档搜索互不干扰
2. **Query vs Chat 必须选对**：Query 只看文档（严格但安全），Chat 混合通用知识（灵活但可能幻觉）
3. **三大可插拔组件**：LLM、Embedding 模型、向量数据库各自独立，可以任意组合本地和云端方案
4. **编辑用户消息会截断历史**：这是一个容易被忽略的行为，修改中间某条消息意味着后面的对话全部重来
5. **Desktop vs Docker 功能差异**：多用户权限、外观定制、嵌入式组件等仅在 Docker 版可用

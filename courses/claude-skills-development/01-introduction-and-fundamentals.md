## 概念讲解

### 1. Skill 是什么

**定义**：Skill 就是一个**文件夹**，里面装着一组指令，告诉 Claude 如何处理特定的任务或工作流。简单说，它是你教会 Claude 做某件事的"知识包"——教一次，以后每次都能用。

**核心思想**：不再每次对话都重新解释你的偏好、流程和领域知识，而是把它们打包成 Skill，让 Claude "学会"一次就够了。

**Skill 文件夹包含什么**：

| 文件/目录 | 是否必须 | 作用 |
|-----------|----------|------|
| `SKILL.md` | **必须** | 主指令文件，Markdown 格式 + YAML frontmatter |
| `scripts/` | 可选 | 可执行脚本（Python、Bash 等） |
| `references/` | 可选 | 按需加载的参考文档 |
| `assets/` | 可选 | 模板、字体、图标等输出资源 |

**适用场景**：
- 从规格文档生成前端设计
- 按固定方法论做调研
- 生成符合团队风格指南的文档
- 编排多步骤流程

> **关键点**：Skill 适合**可重复的工作流**。如果你发现自己每次都在给 Claude 说同样的话，那就该把它变成 Skill。Skill 与 Claude 的内置能力（如 code execution、document creation）天然配合良好。

**本指南的目标受众**：

| 受众 | 说明 |
|------|------|
| **开发者（Developers）** | 希望 Claude 一致地遵循特定工作流 |
| **高级用户（Power Users）** | 希望 Claude 按自己的流程做事 |
| **团队（Teams）** | 希望在组织内标准化 Claude 的使用方式 |

### 2. 核心设计原则

#### 2.1 渐进式披露（Progressive Disclosure）

**定义**：Skill 不是一次性把所有内容塞给 Claude，而是采用**三级加载机制**，按需逐层展开。

**三级体系**：

| 层级 | 加载时机 | 内容 | 位置 |
|------|----------|------|------|
| **第一级** | 始终加载到 Claude 的 system prompt | YAML frontmatter——让 Claude 知道"什么时候该用这个 Skill" | SKILL.md 顶部 |
| **第二级** | Claude 判断 Skill 与当前任务相关时加载 | SKILL.md 正文——完整的指令和指导 | SKILL.md body |
| **第三级** | Claude 按需自行导航发现 | Skill 目录下的其他文件（参考文档、脚本等） | `references/`、`scripts/` 等 |

**为什么重要**：这套机制能**最小化 token 消耗**，同时保持专业能力。Claude 不会被不相关的 Skill 内容淹没——它只在真正需要时才加载详细信息。

> **类比**：想象一本百科全书。第一级是目录（扫一眼就知道有没有你要的），第二级是章节正文（翻到相关章节），第三级是脚注和附录（需要深究时再看）。

#### 2.2 可组合性（Composability）

**定义**：Claude 可以**同时加载多个 Skill**。你的 Skill 应该能与其他 Skill 和谐共处，而不是假设自己是唯一的能力。

**实际含义**：
- 不要在 Skill 中写"你只能做 X"这种排他性指令
- 注意命名和触发条件不要与其他常见 Skill 冲突
- 你的 Skill 可能只负责整个工作流的一部分

#### 2.3 可移植性（Portability）

**定义**：Skill 在 **Claude.ai**、**Claude Code** 和 **API** 上完全一致地工作。创建一次，所有平台都能用。

**限定条件**：前提是运行环境支持 Skill 所需的依赖。比如一个依赖 Python 脚本的 Skill，需要环境中有 Python。

### 3. Skills + MCP 协同

> **注意**：如果你只是构建不涉及 MCP 的独立 Skill，可以跳过这一节，直接看模块二。

#### 3.1 厨房类比（The Kitchen Analogy）

这是理解 Skills 和 MCP 关系的最佳类比：

| 角色 | 类比 | 说明 |
|------|------|------|
| **MCP** | 专业厨房 | 提供工具、食材和设备的**访问权** |
| **Skill** | 菜谱 | 一步步的指导，教你如何**做出有价值的成品** |

两者结合，用户就能完成复杂任务，而不需要自己搞清楚每一步。

#### 3.2 MCP vs Skills 对比

| 维度 | MCP（连接层） | Skills（知识层） |
|------|---------------|-----------------|
| 功能 | 把 Claude 连接到你的服务（Notion、Asana、Linear 等） | 教 Claude **如何有效地使用**你的服务 |
| 提供什么 | 实时数据访问和工具调用 | 工作流和最佳实践 |
| 回答的问题 | Claude **能做什么** | Claude **应该怎么做** |

#### 3.3 没有 Skill vs 有 Skill 的差异

**没有 Skill 时**：
- 用户接了 MCP 但不知道该干嘛
- 客服工单涌入："怎么用你的集成做 X？"
- 每次对话都从零开始
- 结果不一致——不同用户 prompt 不同
- 用户怪你的 connector 不好使，但其实问题在工作流指导缺失

**有 Skill 时**：
- 预设工作流在需要时**自动激活**
- 工具使用一致可靠
- 最佳实践**嵌入每次交互**
- 用户学习曲线更低

> **给 MCP 开发者的建议**：Skill 让你的 MCP 集成更"完整"。当用户在比较不同的 connector 时，有 Skill 的那个能更快地交付价值，形成竞争优势。

### 4. 本指南的两条路径

PDF 在 Introduction 中明确指出了两种读者路径：

| 路径 | 你是谁 | 重点章节 |
|------|--------|----------|
| **构建独立 Skill** | 不涉及 MCP | Fundamentals → Planning & Design → Category 1-2 |
| **增强 MCP 集成** | 已有 MCP server | Skills + MCP 部分 → Category 3 |

两条路径共享相同的技术要求，但你可以只关注与自己用例相关的部分。

**期望产出**：按本指南操作，你能在**一次坐下来的时间内（约 15-30 分钟）**用 skill-creator 构建并测试你的第一个可用 Skill。

## 重点标记

1. **Skill = 文件夹**：核心是 `SKILL.md`，其他（scripts、references、assets）可选
2. **三级渐进式披露**：frontmatter 始终加载 → 正文按需加载 → 附属文件深度探索时加载
3. **可组合 + 可移植**：多 Skill 共存、跨平台一致
4. **MCP 是厨房，Skill 是菜谱**：MCP 给能力，Skill 给方法
5. **有 Skill 的 MCP 集成 >> 没有 Skill 的 MCP 集成**：自动激活、一致结果、更低学习曲线
6. **15-30 分钟可构建首个 Skill**：用 skill-creator 工具

## 概念讲解

### 1. 四大用例类别

Cowork 的设计目标是处理**复杂、多步骤、需要文件访问和长时间执行**的任务。以下是四大类典型用例。

---

#### 1.1 文件与文档管理（File and Document Management）

| 场景 | Prompt 示例 | 说明 |
|------|-------------|------|
| 整理文件 | "Organize my Downloads folder by type and date" | Claude 能一次处理数百个文件，按类型和日期分类到文件夹 |
| 处理收据 | 把收据丢进一个文件夹，让 Claude 生成格式化的费用报告 | 从图片/PDF → 结构化报表 |
| 批量重命名 | 用一致的模式重命名文件，比如 YYYY-MM-DD 格式 | 原来要写脚本的事 |
| 删除重复文件 | 跨目录识别和移除重复文件 | 带删除保护，不会误删 |
| 格式转换 | docx → PDF、图片压缩等 | 在 VM 里直接运行转换工具 |

---

#### 1.2 研究与分析（Research and Analysis）

| 场景 | 说明 |
|------|------|
| **研究综合** | 把网络搜索、文章、论文、笔记的信息合并成连贯的报告或摘要 |
| **会议记录分析** | 从会议记录、访谈、讲座录音中提取主题、要点和行动项 |
| **个人知识综合** | 分析你的笔记、日志或研究文件，发现你可能忽略的模式、主题和关联 |

> **关键价值**：不仅仅是"每篇文档总结一下"，而是**跨文档找到联系和矛盾**——这是人工做起来非常耗时的事。

---

#### 1.3 文档创建（Document Creation）

| 场景 | 说明 | 关键优势 |
|------|------|----------|
| 带公式的表格 | 生成 Excel 文件 | 带 VLOOKUP、条件格式、多 Tab——不是需要修复的 CSV |
| 演示文稿 | 从粗略笔记或会议记录创建幻灯片 | 直接给你 PPT 文件 |
| 从杂乱输入生成报告 | 把语音备忘录和零散笔记变成正式文档 | 从非结构化 → 结构化 |

---

#### 1.4 数据与分析（Data and Analysis）

| 场景 | 说明 |
|------|------|
| **统计分析** | 异常检测、交叉表分析、时间序列分析 |
| **数据可视化** | 用你的数据生成图表 |
| **数据转换** | 清洗、转换和处理数据集 |

### 2. Prompt 设计原则

**核心观点**：使用 Cowork 最大的错误是**把它当聊天机器人用**。正确的方式是**像给同事委派任务一样**。

在委派任何任务之前，先回答三个问题：

| 问题 | 如果你不回答会怎样 |
|------|---------------------|
| **"做完了"长什么样？** | 如果你自己都想不清楚完成态，Claude 也想不清楚 |
| **Claude 需要什么上下文？** | 如果你不说，Claude 会自己假设——有时候对，有时候错 |
| **有什么约束条件？** | Claude 无法读你的心，你没说的限制它不会知道 |

**示例 Prompt 模式**：

```
Read all documents in this folder. Create a synthesis report with:
- Executive summary
- Key themes (with supporting quotes and source files cited)
- Contradictions between sources
- Patterns by [segment/category/source type]
- Specific [requests/recommendations/insights] mentioned
- Questions that remain unanswered
```

> **为什么这个 Prompt 有效**：它明确定义了"完成态"（综合报告），提供了上下文（这个文件夹的文档），指定了约束（具体要包含的 6 个部分）。Claude 不需要猜测。

### 3. 多文档综合分析（Multi-Document Synthesis）

**这是 Cowork 最能节省时间的场景。**

当你有 20+ 个文档（客户访谈、调查结果、工单等），Claude 会：
- **同时阅读所有文档**（并行处理，不是一个个看）
- **跨文档综合分析**——不只是每篇单独总结
- **发现矛盾**：A 文档说 X，B 文档说 Y
- **识别模式**：跨多个来源的共同趋势
- **发现洞察**：人工需要数小时才能识别的关联
- **全程引用来源**：每个论点都标注出自哪个文档

**适用场景**：
- 用户研究：综合 20 份用户访谈
- 市场调研：对比多个来源的数据
- 竞品分析：从多个渠道整合信息
- 学术研究：文献综述

### 4. Sub-Agent 架构详解

Cowork 使用**递归多 Agent 架构（Recursive Multi-Agent Architecture）**来处理复杂任务。

**三层架构**：

```
┌─────────────────────────────────┐
│  Lead Agent (Claude Opus)       │  → 高层任务分解和策略规划
│  任务分析 → 制定计划 → 分配子任务  │
├─────────────────────────────────┤
│  Sub-Agent 1     Sub-Agent 2    │
│  (Sonnet)        (Sonnet)       │  → 并行执行各子任务
│  Sub-Agent 3     Sub-Agent N    │
│  (Sonnet)        (Sonnet)       │
├─────────────────────────────────┤
│  Lead Agent: 结果汇总            │  → 合并结果，质量不达标则触发重试
└─────────────────────────────────┘
```

**工作流程**：
1. **Lead Agent 规划**：Claude Opus 负责高层任务分解和策略
2. **Sub-Agent 并行执行**：多个 Claude Sonnet 实例同时处理各个子任务
3. **结果汇总**：Lead Agent 合并结果，如果某部分不对就触发重试

**关键特性**：
- **自动并行**：Claude 会在合适的时候自动并行化，你不需要显式要求
- **递归分解**：如果子任务本身也很复杂，可以进一步拆解
- **质量把控**：Lead Agent 在汇总阶段会检查质量

> **架构洞察**：Opus 做决策，Sonnet 做执行——这是一个典型的"将军+士兵"模式。Opus 的能力用在最值钱的地方（策略规划），Sonnet 的性价比用在大量并行执行上。

---

### 5. Token 效率与批处理策略

Cowork 任务是计算密集型的，效率很重要：

| 策略 | 说明 |
|------|------|
| **合并相关任务** | 把相关任务放进同一个会话，而不是开多个独立会话 |
| **预先提供所有上下文** | 一次给全，避免来回追问 |
| **明确输出格式** | 指定输出格式、存放位置、命名规范 |
| **简单问题用 Chat** | 不需要文件访问的简单问题，用标准 Chat 就好 |
| **理解用量级别** | 一个 Cowork 会话可能消耗相当于几十条普通 Chat 消息的用量 |

> **核心原则**：Cowork 是为"重活"设计的。简单问题用 Chat，复杂多步骤任务才用 Cowork——这样能最大化你的用量配额。

## 重点标记

1. **别把 Cowork 当聊天机器人**：最大的使用错误就是一来一回地聊，正确方式是像给同事委派任务
2. **多文档综合是杀手级场景**：跨 20+ 文档发现矛盾、模式和洞察，人工需要数小时
3. **三问检查法**：委派前问自己——完成态是什么？需要什么上下文？有什么约束？
4. **Opus 指挥 + Sonnet 执行**：递归多 Agent 架构，高层策略用 Opus，大量并行用 Sonnet
5. **自动并行**：不需要你显式要求，Claude 会自动判断何时并行
6. **用量敏感**：一个 Cowork 会话 ≈ 几十条 Chat 消息的 Token 消耗，简单问题别用 Cowork
7. **合并会话省 Token**：相关任务放同一会话，预先给全上下文，减少来回
